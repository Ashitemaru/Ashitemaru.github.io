---

实验室的东西遇到了点瓶颈，主要是我无法从数据集里面得知什么时候一个 stream 截止。原先采用的方法是切换 channel 就认为 stream 切换，但是用这种方法收集到的数据比较怪异，经常出现时间戳突变（从 0 附近的值突变到七万多秒）。然后今晚我根据 Puffer 数据集的描述，改换为 session ID 变化则认为 stream 切换，结果发现洗出来几万条 trace。

其实这也是符合实际的，不少人去 Puffer player 上也不过就看几秒钟就不看了。然而这会导致 trace 的有效性下降，因为实在是太短了，没有采用的价值。

然而，这个问题似乎并不会影响到 Offline RL 本身，因为 Offline RL 只关注状态的切换，trace 很短并不会影响到状态切换本身。所以我打算用新数据跑一遍训练，顺便检查一下熵的收敛性，明天组会汇报点东西吧。

但是这个服务器也快被我摧残坏了，怎么现在洗数据这么慢。

实验这边稍微做了一下，小规模数据真跳崖了，但是没有像之前一样一下子就下去了，而是每过 10 个 epoch 左右掉下去一点。

整个模型刚起来的时候大概 reward 能打到 15.9 左右，中间有段时间能上到 16.2，但是震荡一段时间后直接就炸了。作为参考，启发式算法 BBA 的 reward 大约是 15.4。总之是炸了。

看 log，感觉是部分地方太过于保守，甚至出现了大量连续选择最低档位的情况。

目前计划用 HYB 以及 MPC 看看 baseline 到底在哪里，如果这两个都打不过 BBA，说明测试脚本还有问题，如果没问题，就是数据集要重新考虑一下，找找黄老师。

至于数据脚本运行太慢的问题，可以试着去 100 核机子上试一下。

贾老师在细节上可能比较想用 Transformer 做，自己比较想去基于之前的一篇文章去做。

目前可能的一个 story 是注意到现在可能视频传输会遇到从 4G 网络且切换到 WiFi 网络的环境变化，现在希望模型能够快速感知到这一变化并作出切换。但是目前对这个 story 的证明还不够，还需要复现这篇论文并且将其迁移到我们原有的工作上。

开题的话，可以尝试去造一些环境切换的 network trace，然后跑 Pensieve 作为 baseline，之后用原先论文的算法击败 Pensieve 即可。

Related work 可以找各种 ABR 的论文，然后总结他们的方法，之后用环境切换的 network trace 跑一遍画图，说明现在他们的表现并不能解决这一问题，并提出自己希望能够提供到何种程度的优化。

baseline 最好有两个，有一个最基本的 Pensieve，另外是特化的 Pensieve，也就是将环境切换的 network trace 也放到训练集里面，naive 地训练一下。后者的必要性在于这是一种最简单的解决方式，如果这种解决方式都能 work，就代表我不需要换到你新提出的方式上了。

目前就这一点的讨论形成了一个待进一步研究的方案，也就是类似于 OS 的设立中期检查和一个面试，只有在前八周或者某个时间段内高强度地完成某一个目标的小组才允许报名申请免考，通过完成拓展功能来代替考试。

这样的话课程组也可以正大光明地控制免考人数，从而免考的人几乎全可以给到 4.0 而不触犯教务处的潜规则。另外，由于这一部分群体毕竟是少数，也可以做到比较好的指导。我的想法是让这一部分人自己提出需求，自己写策划案，自己完成，工作量不能低于一个 baseline 即可。

而免考的要求可以是大作业布置一周内完成一个最基础的功能，就说明这一组整体满足了要求。

另外，总助教也表示了能够接受 4 人左右的额外助教申请，所以下学期的助教基本上也就确定下来了。

现在计划拿着这个方案去找老师们探讨一下，另外在探讨一下习题课究竟以什么样的形式展开就行了。感觉整体已经达到了一个我对软件工程课程的理想状态的期待了。
